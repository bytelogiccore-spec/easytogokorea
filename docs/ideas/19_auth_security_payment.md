## 🔑 1. 사용자 인증 (Authentication)

### 관광객 앱 (외국인)

| 레벨 | 인증 방식 | 사용 가능 기능 |
|------|----------|--------------|
| **0 (비회원)** | 없음 | AR 보기, 길안내, 메뉴판 번역, 교통 정보 |
| **1 (소셜 로그인)** | Google / Apple / 카카오 | AR 낙서 작성, 추천, SNS 공유, 일정 플래너 |
| **2 (여권 인증)** | 소셜 + 여권 OCR | 제로페이 결제, WOWPASS 연동 |
| **3 (보험 연동)** | 소셜 + 여권 + 보험 정보 입력 | 의료 예약, 보험 청구 |

> ⚠️ 한국 전화번호·주민번호 기반 인증은 **외국인에게 불가능**하므로 사용하지 않음

### 사업자 앱 (한국인 소상공인)

| 인증 | 방식 |
|------|------|
| **사업자등록번호** | 국세청 API로 실시간 확인 |
| **한국 전화번호 OTP** | SMS 인증 |
| **의료기관의 경우** | 의료기관 인허가번호 추가 확인 |

### 구현: Firebase Authentication

| 항목 | 내용 |
|------|------|
| 서비스 | Firebase Auth (Spark 무료 플랜) |
| 무료 한도 | 월 5만 MAU |
| 지원 방식 | Google, Apple, 이메일/OTP |
| 서버 연동 | Rust(Axum)에서 Firebase JWT 토큰 검증 |

## 💳 2. 결제 전략 — "우리는 돈을 만지지 않는다"

### 원칙

우리가 직접 결제를 처리하면 **전자금융거래법** 적용 (PG 등록, 자본금 30억+). 따라서 **안내·중개만** 하고, 결제 처리는 전문 사업자에게 위임합니다.

### 결제 수단별 역할

| 결제 수단 | 역할 | 우리가 하는 것 | 돈의 흐름 |
|----------|------|-------------|----------|
| **제로페이 QR** | 매장 결제 (핵심) | 가맹점 안내 + QR 표시 | 사용자 → AlipayPlus → 제로페이 → 매장 |
| **WOWPASS** | 교통 + 카드 결제 | 발급 위치 안내, WebView 임베드 | 사용자 → WOWPASS (직접) |
| **의료관광 예약** | 고액 결제 | 에이전시/병원 연결 | 사용자 → PG사 → 에이전시/병원 |
| **SaaS 구독** | 사업자 월정액 | 직접 수금 (PG 경유) | 사업자 → PG사 → 우리 |

### WOWPASS 연동 방식 (WebView)

```
사용자 → EasyToGo 앱
         → "WOWPASS 구매/충전" 버튼
         → 앱 내 WebView로 WOWPASS 결제 페이지 표시
         → 사용자가 WOWPASS에 직접 결제
         → 완료 콜백 → "충전 완료!" 표시

우리 서버: 돈을 만지지 않음 → 규제 없음
```

### 규제 요약

| 행위 | 규제 | 우리의 선택 |
|------|------|-----------|
| 직접 환전 | 외국환거래법 (자본금 3~10억) | ❌ 안 함 |
| 직접 결제 처리 | 전자금융거래법 (자본금 30억) | ❌ 안 함 |
| 결제 안내·중개 | 규제 없음 | ✅ 이것만 함 |
| SaaS 구독 수금 | PG사 경유 (일반 사업자) | ✅ KG이니시스 등 |

## 🏥 3. 의료 데이터 보안 — "서버에 의료 정보를 저장하지 않는다"

### 적용 법률

| 법률 | 핵심 내용 | 위반 시 |
|------|----------|--------|
| **개인정보보호법** | 민감정보(건강·의료) 별도 동의 필수 | 5년 이하 징역 / 5천만 원 벌금 |
| **의료법 제21조** | 환자 의료 정보 본인 동의 없이 제3자 제공 금지 | 3년 이하 징역 |
| **정보통신망법** | 개인정보 암호화 저장·전송 의무 | 과태료 3천만 원 |
| **GDPR** (EU 환자) | 삭제 요청권 등 | 매출의 4% 과징금 |

### 보안 아키텍처: 온디바이스 + E2E 암호화

```
사용자 폰 (로컬)          우리 서버          보험사/병원
  │                        │                │
  ├─ 영수증 촬영            │                │
  ├─ OCR (온디바이스 AI)    │                │
  ├─ 서류 생성 (로컬 PDF)   │                │
  │                        │                │
  ├─ 앱 내 공유 기능 ──────────────────────→ 이메일/메신저로
  │  (사용자가 직접 전송)    서버에 미저장      본국 보험사에 제출
  │                        │                │
  └─ 폰에 PDF 보관          (관여하지 않음)     보험사가 접수·보관
```

### 데이터 분류

| 저장 ✅ (우리 서버) | 미저장 ❌ (우리 서버) |
|-------------------|-------------------|
| 사용자 ID, 이메일 | 진료 기록, 진단명 |
| 예약 내역 (시간·장소만) | 처방전, 영수증 원본 |
| 앱 사용 로그 | 보험 청구서 내용 |
| 추천·낙서 데이터 | 의료 통역 음성 데이터 |

### 기능별 보안 설계

| 기능 | 처리 위치 | 서버 전송 | 암호화 |
|------|----------|----------|--------|
| **영수증 OCR** | 온디바이스 AI | ❌ | 로컬 처리로 서버 미전송 |
| **보험 서류 생성** | 로컬 PDF 생성 | ❌ | 로컬 저장 후 사용자 직접 전송 |
| **의료 통역** | 온디바이스 AI | ❌ | - |
| **병원 예약** | 에이전시/병원 서버 | 최소 정보만 | TLS 1.3 |

### Rust의 보안 이점

| 특성 | 효과 |
|------|------|
| **메모리 안전성** | 버퍼 오버플로, Use-after-free 등 메모리 취약점 원천 차단 |
| **소유권 시스템** | 민감 데이터가 의도치 않게 복사·유출되는 것을 컴파일 타임에 방지 |
| **제로 코스트 추상화** | 암호화 처리 시 성능 저하 최소화 |
