# HCFA-1500 (CMS-1500) 폼 개요

## 1. 개요
HCFA-1500 양식(현재 공식 명칭: **CMS-1500**)은 미국의 의사, 공급자, 기타 비시설(Non-institutional) 의료 서비스 제공자가 메디케어(Medicare), 메디케이드(Medicaid), 민간 상업 건강 보험 등에 **외래 진료비 및 전문 의료 서비스 비용을 청구할 때 사용하는 표준 건강 보험 청구 양식(Claim Form)**입니다. 

우리나라의 '건강보험 요양급여비용 청구서와 명세서'에 해당하며, 종이 양식도 존재하지만 현재는 시스템상에서 대부분 **837P (Professional)** 형식의 전자 청구(EDI) 데이터 구조로 변환되어 전송됩니다.

## 2. 폼 구조 및 필수 데이터 항목
CMS-1500 폼은 총 33개의 블록(Block)으로 구성되며, 크게 세 가지 주요 파트로 나뉩니다. 향후 글로벌/미국계 보험 청구를 위한 SaaS 데이터베이스를 설계할 때 핵심이 되는 메타데이터들입니다.

### 파트 1: 환자 및 보험 가입자 정보 (Blocks 1-13)
환자의 인구통계학적 정보와 보험 가입자의 정보, 자격 확인에 대한 내용을 수집합니다.

* **Block 1 (보험 적용 유형):** 청구할 보험의 종류 (메디케어, 메디케이드, 일반 그룹 건강 보험 등)
* **Block 1a (가입자 ID 번호):** 보험 카드에 기재된 피보험자의 식별 번호. (수급자 자격 확인을 위해 필수)
* **Block 2 (환자 이름):** 서비스를 받는 환자의 이름.
* **Block 3 & 4 (환자 생년월일, 성별 / 피보험자 이름):** 환자 정보 및 피보험자(환자와 다를 경우) 이름.
* **Block 11 (피보험자의 그룹/정책 번호):** 보험 그룹 식별 번호.
* **Blocks 12 & 13 (서명 항목):** 의료 정보 공개 동의 및 의료비 체당/직접 지급(Assignment of Benefits) 동의 기재란 ("Signature on File" 또는 SOF로 주로 표기).

### 파트 2: 임상 진단 및 서비스 세부 정보 (Blocks 14-24)
제공된 의료 서비스의 내역과 그 '의학적 타당성(Medical Necessity)'을 입증하는 가장 핵심적인 청구 섹션입니다.

* **Block 14 (발병일):** 현재 질병, 부상 또는 임신의 증상 발생일.
* **Block 17 (의뢰 제공자):** 환자를 본 병원으로 보낸 위탁의사(Referring Provider) 이름과 NPI(의사 고유식별번호).
* **Block 21 (진단 코드):** 진료의 의학적 필요성을 설명하는 **ICD-10-CM 진단 코드**. (최대 12개까지 기재 가능하며, 모든 코드는 의무기록으로 증빙해야 함)
* **Block 23 (사전 승인 번호 - Prior Auth):** 보험사의 사전 승인이 필요한 시술이었던 경우 승인 번호 기재.
* **Block 24 (서비스 라인 세부 항목 - 개별 시술 청구내역):**
  * **24A (진료일자 - DOS):** 개별 서비스 수행(시작및 종료) 날짜.
  * **24B (서비스 제공 장소 - POS):** 장소 코드 (외래 11, 병원 21, 원격진료 02 등).
  * **24D (절차/시술 코드 - CPT/HCPCS):** 수행한 시술을 나타내는 5자리 CPT 코드와 보충 설명인 Modifiers.
  * **24E (진단 포인터):** Block 24D에 기재된 행위/시술이 Block 21의 12개 진단 코드 중 어떤 것 때문에 수행되었는지 매핑(A~L). **(이 포인터가 맞지 않으면 보험 삭감의 주요 원인이 됨)**
  * **24F & 24G (청구 금액 및 단위):** 각 시술 항목 당 수가와 단위(횟수, 투약량 등).
  * **24J (수행 진료자 NPI):** 실제 해당 시술을 수행한 임상의의 NPI.

### 파트 3: 의료 제공자 및 청구자 정보 (Blocks 25-33)
청구 병원/클리닉의 식별 정보와 지불 받을 계좌/자격 정보를 기재합니다.

* **Block 25 (연방 세금 ID):** 병원/클리닉의 사업자 세금 ID (EIN) 또는 개인 SSN.
* **Block 27 (할당 수락 - Accept Assignment):** 보험사에서 판정한 허용 금액(Allowed Amount)을 최종 대금으로 수락할지 여부 결정.
* **Block 28 (총 청구액):** 24 파트의 총 합계 금액.
* **Block 32 (서비스 시설 위치):** 실제 의료행위가 발생한 시설의 정보 및 해당 시설의 NPI.
* **Block 33 (청구 제공자 정보):** 실제 비용을 송금받을 주체명과 주소, 결제/청구 NPI.

## 3. EasyToGo B2B SaaS 아키텍처 설계 시 참고(시사점)

작성 중이신 `b2b_saas_billing_architecture.md` 등 B2B 모델 설계 시, 미국계 또는 글로벌 민간 보험을 연동할 가능성이 있다면 아래 사항을 아키텍처에 반영하는 것이 좋습니다.

   * **[클리어링하우스 API 비용 구조 설계 방향]**
     1. **건당 과금 (Per-Transaction) 채택:** Stedi나 pVerify 같은 개발자 친화적 API를 활용해 자격 조회(EDI 270/271) 건당 약 **$0.05 ~ $0.25**의 변동비를 발생시킵니다.
     2. **B2C 과금 (환자 인앱 결제) 전환:** 병원(SaaS)에 API 비용을 청구하는 대신, **외국인 환자가 EasyToGo 앱에서 자신의 보험 자격을 직접 조회하거나 전자 청구(EDI 837)를 보낼 때 소액의 '플랫폼 이용료(Convenience Fee)'를 결제**하도록 설계합니다. (예: 1건당 $1.99 과금)
     3. **전략적 이점:** 
        * **병원 도입 장벽 0%:** 병원 입장에서는 원가나 과금 부담이 아예 발생하지 않으므로 SaaS 도입 저항이 완벽히 사라집니다. (외국인 환자를 받기 위한 허들을 전부 없앰)
        * **환자의 확실한 지불 의사:** 환자는 불안한 타국 병원에서 "내 보험이 커버되는지 확인"하고 "자동으로 청구서류를 생성"해 주는 확실한 편의를 대가로 몇 달러의 이용료를 지불할 강력한 동기(Willingness to Pay)가 있습니다.
2. **데이터 맵핑 (진단명-시술명 연계):**
   * Block 21(진단 코드)과 Block 24E(시술 코드 포인터)의 관계형 설계를 SaaS 데이터베이스 상 식별할 수 있도록(Relation) 구성해야 합니다. 단순 수가 목록 합산이 아닌 "어떤 병 때문에 이 시술을 했는지" 검증이 필요합니다.
3. **청구 기준 거절(Denial) 사전 검증 엔진:**
   * SaaS의 빌링 모듈 내에서 NPI 번호 유효성, CPT-ICD 매핑 오류, Modifier 충돌 등을 캡처해 주는 Validation Layer (사전 거절 방지 로직)이 들어가면 탁월한 제품 차별화 요소가 됩니다.
4. **EDI 837P 지원 모듈 도입:**
   * 장기적으로 종이 폼 출력이 아니라 Clearinghouse(미국의 정보중계소)와 연동해 837P 포맷이나 FHIR 기반 최신 API로 연동할 수 있는 확정성(Adapter 패턴)을 아키텍처단에 열어두시길 권장합니다.
## 4. 해외(한국) 병원의 미국 파트너 연동 시 필드 예외(Foreign Provider Exception)

한국에 위치한 병원(Foreign Provider)이 미국인 환자를 진료하고 미국의 민간 보험원이나 TRICARE 등에 직접 혹은 간접 청구할 때는 **특수한 예외 상황**이 적용됩니다. 본토 기준에 맞춘 모든 항목을 채우기 어렵기 때문입니다.

1. **연방 세금 ID (Block 25) 및 NPI (Block 33a):**
   * 미국 내 사업을 하거나 미국 메디케어 스폰서를 받지 않는 순수 해외 제공자는 원칙적으로 미국의 **세금 ID(EIN/SSN)나 NPI(국가 제공자 식별 번호)**가 없습니다. 
   * **처리 방법:** 결제 대행(Clearinghouse)을 거치거나 국제구호 보험(TRICARE Overseas 등)의 경우, **해외 제공자 전용 식별 코드(Foreign Provider ID)**를 별도 부여받거나, 특수한 더미 값(예: `999999999`)을 넣고 별첨 인보이스를 제출하도록 규칙이 협의됩니다. 
2. **할당 수락 여부 (Block 27 - Accept Assignment):**
   * 미국 내 네트워크 병원(In-network)이 아니므로 기본적으로 **'No(거부)'**에 해당하거나 보험사와의 별도 글로벌 계약 조건에 따릅니다. 환자에게 전액을 받고 환자가 보험사에 '직접 청구(Superbill 방식)'하도록 서류만 떼어주는 경우가 대부분이므로 해당 필드의 중요도가 낮아집니다.
3. **서비스 시설 위치의 우편번호 (Block 32):**
   * CMS-1500은 기본적으로 미국의 5자리(혹은 9자리) ZIP 코드를 요구합니다.
   * **처리 방법:** 한국 주소를 영문으로 적되, ZIP 코드 칸에 한국 우편번호를 넣으면 시스템 오류가 발생할 수 있습니다. 수기 청구(Paper Claim)나 영수증 첨부로 우회하거나, 빌링 SaaS에 **'International Address' 판별 플래그**를 두어 EDI 변환 시 특수 코드(예: `99999`)로 자동 매핑해 주는 예외 처리가 필요합니다.

> **💡 SaaS 설계 인사이트:**  
> 미국 외 병원이 사용하는 B2B SaaS 빌링 모듈에서는 NPI 필수값 검증(Validation)을 우회할 수 있는 **[International Provider Mode] 토글**을 제공하는 것이 핵심 차별화가 될 수 있습니다.
