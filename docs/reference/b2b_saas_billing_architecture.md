# B2B SaaS형 글로벌 의료비 청구 변환 시스템 아키텍처 및 로드맵

## 1. 시스템 아키텍처 (표준 큐 기반 서버 모델)
중앙 집중형 서버에서 다수 병원의 요청을 비동기로 처리하여 안정성과 확장성을 확보합니다.

* **클라이언트 단 (병원 내 설치 프로그램 또는 웹):** 
  원본 데이터 입력 및 데이터 비식별화 전처리를 수행합니다. 서버에서 반환받은 의학 데이터와 로컬에 남겨둔 환자 정보를 재결합하여 최종 영수증을 렌더링합니다.
* **API 게이트웨이 & 큐(Queue) 서버:** 
  대량의 API 요청을 수신하고 인증 처리를 합니다. Message Broker(예: Redis, RabbitMQ)를 사용하여 요청을 대기열(Queue)에 적재합니다. (Rust의 Actix-web이나 Python FastAPI 활용)
* **워커 노드 (Worker Nodes):** 
  대기열에서 작업을 꺼내어 실제 변환을 수행하는 독립된 서버군입니다. 구동 중인 LLM 파이프라인(텍스트 정제 및 엔티티 추출)과 고성능 매핑 데이터베이스(Rust 기반)를 거쳐 결과값을 생성 후 반환합니다.

## 2. 핵심 로직: 데이터 비식별화 전처리 (Client-Side)
환자의 민감한 의료 정보(PHI)가 외부 망(서비스 서버)으로 유출되는 것을 원천 차단하기 위해, 병원 측 프로그램에서 전송 직전에 실행되는 필수 로직입니다.

* **1단계 (식별자 마스킹 및 토큰화):** 
  환자의 이름, 생년월일, 주민등록번호, 전화번호 등을 고유한 임시 난수 토큰(예: `Req_20260309_A1B2`)으로 치환합니다.
* **2단계 (순수 의료 정보 분리 전송):** 
  서버로는 식별이 불가능한 상태의 순수 의학 데이터(임시 토큰 + KCD 진단명 + 의무기록 텍스트 + EDI 코드)만 암호화(HTTPS/TLS)하여 전송합니다. 이때 병원 로컬 DB에는 `[임시 토큰 ↔ 실제 환자 정보]` 매핑 테이블이 안전하게 보관됩니다.
* **3단계 (클라이언트 재결합 - Re-identification):** 
  서버가 임시 토큰과 함께 영문 변환 및 매핑 결과(JSON)를 반환하면, 클라이언트 프로그램이 로컬 매핑 테이블을 참조해 원래 환자 정보를 씌워 최종 문서를 완성합니다.

## 3. 다단계 데이터 매핑 워커 로직 (Server-Side)
워커 노드 내부에서 실행되는 텍스트 분석 및 코드 변환 파이프라인입니다.

* **Step A (LLM 텍스트 추출):** 전달받은 비식별화된 의무기록 텍스트에서 약어를 정제하고 '처치/수술명' 엔티티만 분리 추출.
* **Step B (국제 표준 1차 매핑):** KCD 진단명을 ICD-10으로 변환하고, 추출된 처치명을 영문 표준 의학 용어(SNOMED-CT 등)로 1차 매핑.
* **Step C (국가별 맞춤 2차 매핑):** 사용자가 선택한 타겟 국가(예: 미국)에 맞춰 영문 수술명을 CPT 코드 등으로 2차 고속 매핑 (Rust 엔진 활용).

## 4. 핵심 로직: 국가별 동적 매핑 및 라우팅 파이프라인
클라이언트가 API를 호출할 때 `target_country` (제출 국가) 파라미터를 함께 전송하면, 워커 노드와 클라이언트가 조건에 맞춰 다르게 동작합니다.

**공통 처리 로직 (Server-Side):**
* KCD → ICD-10 (글로벌 공통 진단 코드) 자동 변환.
* 비식별화된 의무기록 텍스트에서 '처치/검사명'을 추출하여 국제 표준 의학 용어(SNOMED-CT 등)로 1차 정제.

**분기 1: 미국 (US) 전용 파이프라인**
* **서버 역할:** 1차 정제된 표준 용어를 Rust 매핑 엔진에 태워 CPT 코드로 2차 고속 매핑 후 반환.
* **클라이언트 역할:** 반환된 CPT 코드와 환자 정보를 결합하여, 미국 사보험사 및 메디케어 표준 청구 양식인 **HCFA-1500 폼**에 맞춰 PDF 렌더링.

**분기 2: 유럽 및 중국 (EU / CN) 파이프라인**
* **서버 역할:** 복잡한 CPT 매핑을 생략하고, 정확하게 번역된 영문(또는 중문) 상세 진료 내역과 시술명만 깔끔하게 반환.
* **클라이언트 역할:** 글로벌 사보험사(AXA, Allianz 등) 심사역이 보기 편한 **영문 상세 영수증(Itemized Bill)** 형태로 데이터 배치 및 렌더링.

**분기 3: 일본 (JP) 전용 파이프라인**
* **서버 역할:** 추출된 진료 내역을 일본 독자 규격에 맞는 일본어 의료 용어로 번역하여 반환.
* **클라이언트 역할:** 일본 건강보험조합의 '해외요양비' 청구 전용 법정 양식인 **진료내용명세서(A4 양식)**에 데이터를 매핑하여 렌더링.

## 5. 단계별 개발 마일스톤 (업데이트)
* **Phase 1: 비식별화 클라이언트 모듈 및 API/큐 서버 세팅**
  * 정규표현식 및 룰베이스를 활용한 클라이언트 측 환자 정보 마스킹/토큰화 모듈 개발.
  * 서버의 API 게이트웨이 및 Message Broker 환경 구축.
* **Phase 2: 서버 사이드 매핑 데이터베이스 및 국가별 분기 로직 구축**
  * Rust 기반 고성능 KCD ↔ ICD-10 ↔ CPT 인메모리 데이터베이스 구축.
  * API 수신 시 `target_country` 파라미터에 따른 라우터(Router) 구현.
* **Phase 3: 의료 도메인 특화 모델 미세조정 (Fine-tuning)**
  * 비식별화 처리된 가상의 진료기록 데이터셋 500~1,000건 구축.
  * 서버 GPU 환경에 텍스트 추출용 LLM 셋업 및 워커용 모델 학습(추출 정확도 향상).
* **Phase 4: B2B 시스템 통합 및 국가별 템플릿 렌더링**
  * 클라이언트 쪽에서 수신된 데이터를 국가별 렌더링 템플릿(미국 HCFA-1500, 일본 법정 양식, 글로벌 Itemized Bill)에 탑재.
  * 국가별 청구 폼을 PDF로 자동 생성하는 렌더링 로직 통합.
